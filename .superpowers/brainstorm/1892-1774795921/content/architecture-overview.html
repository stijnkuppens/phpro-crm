Architecture Overview

Full data flow: from "Exporteren" click to file download

System Architecture — Export Jobs
1. User clicks
"Exporteren → Excel"
on Accounts page
→
2. Server Action
INSERT into jobs table
+ pgmq.send('export_jobs')
→ returns job ID to client
→
3. toast + redirect
"Export gestart"
optional: navigate to /admin/jobs
4. pg_cron (1 min)
Calls Edge Function
via pg_net.http_post()
→
5. Edge Function
a. pgmq.read(vt=120)
b. Query all matching rows
c. Generate XLSX or CSV
d. Upload to documents bucket
e. UPDATE jobs → completed
f. INSERT notification
g. pgmq.archive()
6. Realtime
jobs table UPDATE
→ client sees status change
+ progress %
→
7. Notification
🔔 Bell shows
"Export Accounts gereed"
links to job detail
→
8. Download
Click notification or
job detail modal
→ signed URL download

Database Tables

jobs (new table)
id            uuid PK
type          text ('export')
status        text (pending → processing → completed | failed)
entity        text ('accounts')
format        text ('xlsx' | 'csv')
filters       jsonb (orFilter, eqFilters, sort)
progress      int (0-100)
row_count     int
file_path     text (storage path)
file_size     bigint (bytes)
error         text
requested_by  uuid FK → auth.users
created_at    timestamptz
started_at    timestamptz
completed_at  timestamptz
pgmq queue: export_jobs
msg_id      bigint (auto)
read_ct     int (retry count)
enqueued_at timestamptz
vt          timestamptz
message     jsonb {
  job_id: uuid
}

Archived messages go to
pgmq.a_export_jobs

Key Design Decisions

✓ jobs table = source of truth for UI, pgmq queue = processing mechanism only
✓ Filters stored as JSONB — exact same shape as useEntity's fetchList params
✓ documents bucket reused — exports stored at exports/{job_id}/export.{ext}
✓ RLS on jobs — users see only their own jobs
✓ Realtime on jobs table — already in supabase_realtime publication pattern
✓ Dead letter after 3 retries — if read_ct > 3, mark job as failed
